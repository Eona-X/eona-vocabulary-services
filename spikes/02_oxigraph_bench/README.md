# Spike 2 — Oxigraph vs Fuseki benchmark

**Gates:** L1/L2 swap (ADR-004 §1).

## What this spike answers

ADR-004 requires "bulk-load and query-mix benchmark on a representative
dataset. Output: RSS, p50/p95 latency, load throughput vs. Jena/Fuseki."

No hub-owned representative dataset exists. The harness therefore runs
both engines under identical conditions against two public-reference
workloads:

- `inputs/bulk-skos.nt` — 100 000-triple deterministic SKOS taxonomy
  generated by Spike 1's `gen_bulk.py` (re-used here verbatim;
  symlinked at run time so its hash is stable across spikes).
- `inputs/queries/q*.rq` — 8 SPARQL queries spanning the operations
  expected in a Vocabulary Hub: prefix label lookup, broader/narrower
  traversal, concept-scheme enumeration, content-negotiated CONSTRUCT,
  property-path closure, named-graph filter, UPDATE INSERT, ASK.

The manifest records `inputs.kind = "public-reference"`. When the hub
team produces a real workload, dropping the files into `inputs/` and
re-running yields a manifest with `inputs.kind = "hub"` and the same
artefact layout.

## Metrics captured

For each (engine × workload) pair:

- `load_throughput_triples_per_sec` — wall time of the bulk POST.
- `query_latency.json` — per-query n, min, p50, p95, p99, max, mean.
- `rss_<engine>.csv` — host-RSS time series of the container's main
  process (sampled at 4 Hz via `_lib/rss.py` reading
  `/proc/<docker-inspect-pid>/status`).
- `peak_rss_mb` — max of the trace, reported in `summary.md`.

Each query is run **warm** (3 untimed primes followed by 50 timed
iterations) to make p95 meaningful on a small sample.

## Outputs

```
results/<run-id>/
├── manifest.json
├── load_results.json
├── query_latency.json
├── rss_oxigraph.csv
├── rss_fuseki.csv
├── raw/q<NN>/<engine>.<iter>.txt   # full body of every timed query response
└── summary.md
```

The per-iteration response bodies are kept so the same answers can be
diffed across engines later — if Oxigraph's p50 is great but it returns
different bindings than Fuseki on `q03`, that's a coverage bug masked as
a performance win.

## Variants

Spike 2 runs in two compose profiles, both using the engines' public
binaries. Pick the one you want to characterise:

| `VARIANT=` | Fuseki backend | Oxigraph backend | What it measures |
|---|---|---|---|
| `tmpfs` (default) | `--mem` (in-process) | RocksDB on tmpfs | engine-only — what's the engine cost without storage I/O? |
| `disk` | TDB2 on a Docker volume | RocksDB on a Docker volume | deployment-realistic — what does the hub actually run? |

The variant name is recorded in every manifest, summary header, and
results path, so future readers can never mistake one for the other.

## Re-running

```bash
VARIANT=tmpfs ./run.sh        # engine-only
VARIANT=disk  ./run.sh        # deployment-realistic
```

`run.sh` does `docker compose down` between variants so the shared
container names / ports don't collide.
